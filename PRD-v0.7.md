## 1. 핵심 개념 — 사용자(User)와 역할(Role)의 분리

### 1.1 사용자(User) — 글로벌·평등
- 로그인한 모든 사람은 동등한 사용자
- 사용자 자체에는 권한 등급이 없음
- 누구나 새 앨범 생성 가능, 누구나 초대받은 앨범에 참여 가능

### 1.2 역할(Role) — 앨범 단위로 부여
- 한 사용자는 앨범마다 다른 역할을 가질 수 있음
- 앨범을 생성한 사용자는 **그 앨범의** 방장(owner)이 됨
- 역할은 `album_members` 테이블에 (album_id, user_id) 조합별로 저장

### 1.3 시나리오 예시

```
유저 A, B, C, D (모두 평등한 사용자)

  A가 "제주 가족여행" 앨범 생성 → A=방장, B·C·D=멤버 초대
  B가 "친구들 부산여행" 앨범 생성 → B=방장, A·C=멤버 초대

  결과: A는 앨범1에서 방장, 앨범2에서 멤버
        B는 앨범1에서 멤버, 앨범2에서 방장
```

### 1.4 UI 영향
- 홈(S-03) 앨범 카드에 **내 역할 배지** 표시 (`방장` / `Co-host` / `멤버`)
- 글로벌 표현 금지: "당신은 방장입니다" ❌
- 컨텍스트 표현 필수: "이 앨범에서 당신은 방장입니다" ✅

## 2. 확정 정책 전체

### 2.1 인증 정책
- MVP는 **Google 로그인만** 지원
- Kakao·Naver 로그인은 PWA/Safari 호환성 검증 후 Post-MVP 추가
- 공개 정보: 닉네임, 프로필 이미지 (이메일·연락처 비공개)

### 2.2 프로필 초기 설정 정책 (S-02b)
- 첫 로그인 직후 프로필 초기 설정 화면을 강제 진입
- Google 프로필 데이터(이름·아바타)를 기본값으로 자동 채움
- 사용자가 그대로 사용하거나 변경 가능 ("시작하기" 한 번이면 완료)
- 이후 재로그인 시 자동 스킵 (홈으로 직행)
- 가입 후 언제든 S-12 프로필 설정에서 수정 가능

**프로필 데이터 규칙**

| 항목 | 규칙 |
|---|---|
| 닉네임 | 필수, 2~20자, 앞뒤 공백 trim, 중복 허용 |
| 프로필 이미지 | 선택, JPG/PNG/WEBP, 5MB 이내 |
| 프로필 이미지 리사이즈 | 업로드 시 256×256으로 자동 변환 |
| 프로필 이미지 저장소 | Cloudflare R2 별도 버킷 `user-profiles/{user_id}.{ext}` |
| 앨범 5GB 한도 영향 | 없음 (앨범 용량 산정에서 제외) |

### 2.3 앨범 내 역할 구조

| 역할 | 일정 관리 | 멤버 관리 | 앨범 삭제 | 초대 링크 | 사진·일기 |
|---|---|---|---|---|---|
| 방장 (owner) | ✅ | ✅ | ✅ | ✅ | ✅ |
| 공동 관리자 (co_host) | ✅ | ✅ (강퇴 제외) | ❌ | ✅ | ✅ |
| 멤버 (member) | ❌ | ❌ | ❌ | ❌ | ✅ |

- 앨범 생성자가 자동으로 그 앨범의 방장이 됨
- 방장은 같은 앨범의 다른 멤버를 Co-host로 지정 가능
- **Co-host 인원 상한**: 무제한 (가족·지인 그룹 자연 제한)
- **앨범당 멤버 인원 상한**: 20명 (5GB ÷ 20명 ≈ 1인당 250MB 합리적 분배)

### 2.4 방장 자동 승계 정책

방장 자리가 비는 상황과 처리 절차:

| 트리거 | 처리 |
|---|---|
| 방장이 앨범에서 탈퇴 시도 | UI에서 차단. 대신 "방장 위임 후 탈퇴" 또는 "앨범 삭제 요청"만 허용 |
| 방장의 계정 탈퇴 | 자동 승계 절차 진행 |
| 자동 승계 우선순위 | ① Co-host 중 가장 먼저 지정된 사람 → ② Co-host 없으면 가장 먼저 가입한 일반 멤버 → ③ 멤버도 없으면 앨범 자동 삭제 |

### 2.5 앨범 삭제 정책
- 방장의 삭제 요청 → **7일 유예 기간** → 영구 삭제
- 유예 기간 중 모든 멤버에게 인앱 알림 + 앨범 진입 시 상단 경고 배너 표시
- 유예 기간 중 방장은 삭제 취소 가능
- 유예 기간 중 멤버는 사진 다운로드 가능
- 자동 삭제 구현: **Supabase `pg_cron`** 으로 매일 새벽 1회 스케줄 실행 → 7일 경과 앨범 검색 → R2 사진 일괄 삭제 → DB CASCADE 삭제

### 2.6 초대 링크 정책
- **7일 유효**, 만료 후 자동 비활성화
- 앨범당 **활성 링크 1개만** 유지 (새 링크 발급 시 기존 자동 만료)
- 별도 승인 절차 없음 (아는 사람끼리 공유 전제)
- 링크 발급 권한: 방장 + Co-host

### 2.7 강퇴 / 탈퇴 정책

| 상황 | 콘텐츠 처리 | UX 안내 |
|---|---|---|
| 방장·Co-host가 멤버를 **강퇴** | 해당 멤버의 사진·감상문 **즉시 삭제** | 강퇴 직전 "이 멤버의 콘텐츠가 모두 삭제됩니다" 확인 모달 |
| 멤버 본인이 앨범 **탈퇴** | 콘텐츠 **유지** | 탈퇴 전 "탈퇴해도 남긴 사진과 글은 삭제되지 않습니다" 안내 |

### 2.8 계정 탈퇴 vs 앨범 탈퇴 분리

| 구분 | 범위 | 처리 |
|---|---|---|
| 앨범 탈퇴 (Leave Album) | 한 앨범만 빠짐 | 다른 앨범 영향 없음, 계정 유지 |
| 계정 탈퇴 (Withdraw) | 서비스 전체 탈회 | 참여 중인 모든 앨범에서 자동 탈퇴 + `users` 레코드 삭제 + 본인 사진·일기 CASCADE 삭제 + 본인이 owner인 앨범은 방장 자동 승계(§2.4) 후 본인만 빠짐 |

### 2.9 탈퇴 멤버 잔존 콘텐츠 표시

- **앨범 탈퇴 후 콘텐츠 유지**: 작성자 닉네임·아바타 그대로 표시
- **계정 탈퇴 후**: `users` 레코드 부재 → 표시 시 닉네임 자리에 `(탈퇴한 사용자)`, 기본 아바타 사용
  - SQL 조회 시 `LEFT JOIN users` + `COALESCE` 처리

### 2.10 사진 저장 정책

| 항목 | 규칙 |
|---|---|
| 앨범당 최대 용량 | **5 GB** |
| 동영상 | 미지원 (Post-MVP 검토) |
| 파일 형식 | JPG, JPEG, PNG, HEIC, WEBP |
| 파일당 최대 크기 | 20 MB (리사이즈 전 원본 기준) |
| 1회 업로드 최대 장수 | 10장 |
| 클라이언트 리사이즈 | 최장변 2,560px 자동 압축 (browser-image-compression) |
| 커버 이미지 | 앨범 5GB 한도 **포함**, 최장변 1,280px 리사이즈 |
| 프로필 이미지 | 앨범 5GB 한도 **제외** (§2.2 참조) |
| 저장소 | Cloudflare R2, 객체 키 `albums/{album_id}/{photo_id}.{ext}` |
| 용량 검증 시점 | ① 클라이언트 (UX용 즉시 차단) ② Edge Function (서버 신뢰 게이트, 진짜 검증) |
| 용량 차감 동시성 | DB 트리거로 원자 연산 (§5.2 참조) |

### 2.11 날씨 기록 정책

여행이 끝난 뒤에도 그날 실제 날씨를 변함없이 기록으로 남기기 위함.

**저장 데이터 (테이블: `daily_weather`)**

| 컬럼 | 의미 |
|---|---|
| `temp_max` | 최고 기온 (°C, 소수점 1자리) |
| `temp_min` | 최저 기온 (°C, 소수점 1자리) |
| `weather_code` | 5단계 분류 (아래 표) |
| `is_forecast` | 예보(true) / 실측(false) |
| `fetched_at` | 마지막 API 조회 시각 |

**날씨 5단계 분류**

| 코드 | 분류 | 아이콘 | Open-Meteo WMO 매핑 |
|---|---|---|---|
| 0 | 맑음 | ☀️ | 0 |
| 1 | 구름 조금 | ⛅ | 1, 2 |
| 2 | 흐림 | ☁️ | 3, 45, 48 |
| 3 | 비 | 🌧️ | 51-67, 80-82, 95-99 |
| 4 | 눈 | ❄️ | 71-77, 85-86 |

**갱신 정책 (날짜 기준)**

| 날짜 구분 | 정책 | API 엔드포인트 |
|---|---|---|
| 과거 (오늘 이전) | 1회 fetch 후 영구 고정 (`is_forecast=false`) | `archive-api.open-meteo.com/v1/archive` |
| 오늘 / 미래 16일 이내 | 6시간 stale 후 재조회 (`is_forecast=true`) | `api.open-meteo.com/v1/forecast` |
| 미래 17일+ | 조회 불가, UI에서 날씨 영역 숨김 | — |

**좌표 미설정 시 처리**: `albums.destination_lat/lng`가 NULL이면 날씨 영역 graceful hide (다른 기능은 정상 작동)

**조회 흐름**: 클라이언트가 직접 Open-Meteo를 호출하지 않고 **Supabase Edge Function `fetch-weather`** 경유. 함수는 호출자의 Supabase JWT를 검증하고, 해당 앨범의 멤버인 경우에만 실행 (RLS 강제). WMO 코드 → 5단계 압축은 함수 내부에서 처리.

### 2.12 목적지 좌표 입력 정책

- 사용자는 앨범 생성 시 **목적지 텍스트** 입력 (예: "제주도", "도쿄")
- 서버사이드 **자동 지오코딩** (Open-Meteo Geocoding API, 무료) → 좌표(`destination_lat/lng`) 자동 변환·저장
- 지오코딩 실패 시 좌표 NULL로 저장 → 날씨 기능 비활성화, 다른 기능은 정상
- 사용자가 별도 지도 선택 UI 없이도 사용 가능 (MVP 단순화)

### 2.13 알림 메커니즘 (MVP)

| 알림 유형 | 구현 방식 |
|---|---|
| 앨범 삭제 예정 | **인앱 배너** (앨범 진입 시 상단에 D-Day 표시) + 홈 화면 카드 상단 빨간 배너 |
| 멤버 강퇴 | 강퇴된 멤버는 해당 앨범 진입 시도 시 "더 이상 참여할 수 없습니다" 안내 |
| 멤버 합류 | 인앱 알림 없음 (멤버 목록 확인으로 대체) |
| 푸시 알림 | **Post-MVP** (PWA Web Push 호환성 검증 필요) |
| 이메일 알림 | MVP 미지원 |

### 2.14 타임존 정책

- 시스템 시각(`created_at`, `fetched_at` 등): **UTC 저장**
- 사용자 표시 시각: **Asia/Seoul (KST) 기준 변환**
- 날씨 API 호출 시 좌표 기준 현지 시각 사용 (`timezone=auto` 파라미터)
- 일정 시간(`scheduled_time`): 사용자가 입력한 텍스트 시각 그대로 (현지 시간 의도로 간주)

## 5. 데이터 모델

### 5.1 테이블 일람

| 테이블 | 용도 | PK |
|---|---|---|
| `users` | 사용자 프로필 | `id` (auth.users 참조) |
| `albums` | 몽글여행 본체 | `id` |
| `album_members` | 앨범별 멤버 + 역할 | `id`, UNIQUE(album_id, user_id) |
| `album_invites` | 초대 링크 | `id`, UNIQUE active per album |
| `itinerary_items` | 일자별 일정 항목 | `id` |
| `photos` | 사진 메타데이터 | `id` |
| `diary_entries` | 감상문 | `id`, UNIQUE(album_id, user_id, date) |
| `daily_weather` | 날짜별 날씨 기록 | (album_id, date) 복합 |

### 5.2 핵심 제약·트리거

- `album_members` role ENUM: `'owner' | 'co_host' | 'member'`
- `album_invites` 부분 유니크 인덱스: 앨범당 활성(`is_active=true`) 링크 1개만 허용
- `diary_entries` UNIQUE(album_id, user_id, date): 1인 1일 1개 강제
- `photos` INSERT/DELETE 트리거: `albums.storage_used_bytes`를 원자적으로 증감
- ON DELETE CASCADE: 앨범 삭제 시 모든 종속 데이터 자동 삭제

### 5.3 핵심 컬럼 메모

- `albums.destination_lat`, `destination_lng`: numeric(9,6), nullable — 지오코딩 결과
- `albums.storage_limit_bytes`: bigint, default 5368709120 (5 GB)
- `albums.delete_requested_at`: timestamptz, nullable — 7일 유예 시작 시각
- 자세한 DDL은 별도 파일 `schema.sql` 참조
